# 自动化框架 demo 调试层

`demo.py` 不是正式框架执行入口，而是一个单接口调试脚本。

## 1. 它在做什么

这个脚本手动写了两段请求数据：

- 登录接口
- 上传接口

然后执行流程是：

1. 先调用登录接口
2. 从登录响应里拿到 `token`
3. 再调用上传接口

这本质上是在手工验证“前置接口取值 + 后置接口使用”的链路。

## 2. 为什么自动化框架里会有这种脚本

因为在正式把接口抽象进 Excel 驱动框架之前，开发或测试通常会先做单接口验证：

- 接口通不通
- 参数格式对不对
- token 能不能拿到
- 上传接口是否依赖登录态

`demo.py` 就是这种“预验证脚本”。

## 3. 它和正式框架的关系

可以理解成：

- `demo.py`：手工版验证流程
- `test_runner.py`：通用版框架流程

也就是说，这个脚本像是在给后面的自动化框架打样。

## 4. 当前脚本的一个明显问题

脚本里虽然拿到了：

```python
token = res1.json()["data"]["token"]
```

但上传请求的 `headers` 仍然是 `None`，没有真正把 token 带到上传接口里。

这说明它更像一个教学中的中间版本，而不是完整成品。

真正要调用登录态上传接口，通常还需要：

```python
upload_data["headers"] = {"Authorization": token}
```

## 5. 一句话总结

`demo.py` 属于自动化框架里的调试层或验证层，主要用于先手工跑通接口链路，再逐步抽象进正式框架。
