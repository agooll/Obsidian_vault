# 自动化框架 data 数据层

`data` 目录的职责是存放测试数据源，这个示例里使用的是 Excel 文件 `测试用例.xlsx`。

## 1. 为什么 `data` 层重要

自动化框架里，测试代码和测试数据最好分离：

- 测试代码负责“怎么执行”
- 测试数据负责“执行什么”

这样同一套执行代码就可以跑很多条不同用例。

## 2. 这份 Excel 的结构

这个文件的 `Sheet1` 采用了两行表头设计：

第一行是中文说明：

```text
编号, 模块, 场景, 标题, 请求方式, 路径, 请求头, url参数, data参数, json参数, 文件参数, 校验字段, 预期结果, 数据库校验内容, 数据库预期结果, json提取, sql提取, 是否执行
```

第二行是程序真正读取的字段名：

```text
id, feature, story, title, method, path, headers, params, data, json, files, check, expected, sql_check, sql_expected, jsonExData, sqlExData, is_true
```

这里的设计很实用：

- 第一行方便测试人员阅读
- 第二行方便程序直接按 key 取值

## 3. 每一列分别代表什么

- `feature`：模块名，对应 allure 的大功能模块
- `story`：场景名，用于细分业务场景
- `title`：测试用例标题
- `method`：请求方式，如 `get`、`post`
- `path`：接口路径
- `headers`：请求头
- `params`：URL 查询参数
- `data`：表单数据
- `json`：JSON 请求体
- `files`：文件上传参数
- `check`：jsonpath 断言表达式
- `expected`：预期结果
- `sql_check`：数据库校验 SQL
- `sql_expected`：数据库预期结果
- `jsonExData`：从接口响应中提取数据
- `sqlExData`：从数据库中提取数据
- `is_true`：是否执行当前用例

## 4. 这份 Excel 已经覆盖的场景

当前有效用例主要有：

- 管理员登录成功
- 未登录查询用户列表失败
- 带 token 查询用户列表成功
- 图片上传成功

这说明它已经完整演示了：

- 登录态接口关联
- 未登录与已登录场景对比
- 文件上传场景

## 5. 为什么这叫数据驱动

因为测试函数本身不关心“具体测哪个接口”，它只负责：

- 读取当前行数据
- 组装请求
- 发起调用
- 校验结果

至于要测什么，都由 Excel 决定。

所以新增测试用例时，很多情况下只需要“加一行 Excel”，而不用改测试代码。

## 6. 这个数据层的优点

- 入门成本低，适合教学和小项目
- 测试人员容易维护
- 字段含义清晰
- 能直观看到请求、断言、提取、关联

## 7. 这个数据层的局限

- Excel 合并多人维护时容易冲突
- 复杂嵌套数据在单元格里可读性一般
- 字符串形式的字典要靠 `eval()` 转回对象，不够安全
- 用例规模一大，查找和复用成本会上升

## 8. 一句话总结

`data` 层就是自动化框架的“数据输入层”，负责把测试场景以结构化方式交给执行层，是数据驱动框架的起点。
