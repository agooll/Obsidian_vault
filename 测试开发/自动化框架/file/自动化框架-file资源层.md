# 自动化框架 file 资源层

`file` 目录里放的是接口自动化执行时要用到的静态资源。当前项目里放的是：

```text
1.jpg
```

## 1. 这个目录的作用

它主要服务于文件上传类接口。

例如当前 Excel 里有一条上传图片用例：

```python
{"file": ("1.jpg", open("./file/1.jpg", "rb"), "jpg")}
```

接口执行时会读取 `file/1.jpg`，并把它作为 multipart/form-data 的文件字段上传。

## 2. 为什么资源要单独放一层

如果把文件路径写散在代码里，会有几个问题：

- 上传素材不好统一管理
- 项目迁移时路径容易出错
- 测试数据和测试资源耦合太紧

单独建 `file` 目录后，职责会更清晰：

- `data` 层管理测试数据
- `file` 层管理测试资源
- `testcases` 层只负责调用

## 3. 这层在自动化框架里的意义

很多接口自动化不只是普通的 `get/post`：

- 上传图片
- 上传附件
- 导入 Excel
- 导入 CSV
- 批量导入模板

这类场景都需要一个专门的资源目录。

## 4. 当前写法的注意点

示例里直接在 Excel 或脚本里写：

```python
open("./file/1.jpg", "rb")
```

这种写法能跑，但工程化上有几点风险：

- 文件句柄没有统一关闭管理
- 相对路径依赖当前运行目录
- Excel 中直接写 `open()` 表达式，对维护和安全都不够友好

更合理的方式通常是：

- Excel 只写文件名或相对路径
- 代码里统一拼接资源目录
- 代码里统一打开并关闭文件

## 5. 一句话总结

`file` 层本质上是自动化框架的“测试资源层”，主要为上传、导入等接口场景提供文件依赖。
