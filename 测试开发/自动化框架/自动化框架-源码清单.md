# 5.10 源码清单

这篇笔记专门对应目录 `D:\测试开发\1.基础\5.10`，用于快速说明每个源码文件在自动化框架里的职责。

## 1. 根目录文件

### `run.py`

职责：

- 统一启动 pytest
- 生成 allure 原始结果
- 生成 allure HTML 报告

定位：

- 入口层

### `pytest.ini`

职责：

- 配置 pytest 日志格式
- 配置日志落盘位置
- 配置控制台日志输出

定位：

- 配置层

### `demo.py`

职责：

- 手工调试登录接口
- 手工调试上传接口
- 验证 token 获取链路

定位：

- 调试层

## 2. 目录文件

### `data/测试用例.xlsx`

职责：

- 存放接口自动化测试用例
- 定义请求参数、断言、提取、数据库校验
- 通过 `is_true` 控制是否执行

定位：

- 数据层

### `utils/excel_utils.py`

职责：

- 读取 Excel
- 读取第二行字段名
- 从第三行开始取测试数据
- 组装成 `list[dict]`
- 过滤 `is_true` 为假的用例

定位：

- 工具层

### `testcases/test_runner.py`

职责：

- 通过 pytest 参数化执行 Excel 用例
- 组装请求
- 发起接口调用
- 做 HTTP 断言和数据库断言
- 做 JSON 提取和 SQL 提取
- 维护全局变量池完成接口关联

定位：

- 执行层

### `file/1.jpg`

职责：

- 提供上传接口所需的测试文件

定位：

- 资源层

## 3. 这个框架的最小闭环

从源码清单看，它的执行逻辑可以压缩成一句话：

> Excel 定义用例，`excel_utils.py` 读取数据，`test_runner.py` 统一执行请求和断言，`run.py` 负责启动并产出报告。

## 4. 为什么这份清单有价值

学习自动化框架时，最容易混淆的是：

- 哪些文件负责数据
- 哪些文件负责执行
- 哪些文件只是调试辅助

把源码职责拆清后，后面再看框架扩展就容易很多，比如：

- 配置抽离
- 请求封装
- 数据库封装
- 报告增强

## 5. 一句话总结

`5.10` 这个目录虽然小，但已经具备一个接口自动化框架的最小骨架，源码职责划分也比较清晰，适合拿来做入门理解模板。
