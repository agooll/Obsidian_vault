# 5.10 自动化框架解析

这个示例项目是一个典型的 Excel 驱动接口自动化框架，核心链路如下：

```text
Excel测试数据
    ↓
utils/excel_utils.py 读取并过滤用例
    ↓
pytest 参数化把每一行用例注入 test_case
    ↓
requests.request 发起接口请求
    ↓
HTTP断言 / 数据库断言
    ↓
json提取 / sql提取 写回全局变量池
    ↓
后续用例通过 Jinja2 模板引用前置提取值
    ↓
run.py 统一执行 pytest 并生成 allure 报告
```

## 1. 目录职责拆分

- `data`：存放 Excel 测试用例，是整个框架的数据驱动入口
- `utils`：封装公共工具，这里主要负责读取 Excel
- `testcases`：真正的执行层，负责请求、断言、提取
- `file`：存放上传类接口依赖的静态资源
- `pytest.ini`：统一配置日志输出
- `run.py`：统一启动 pytest 和 allure 报告生成
- `demo.py`：单接口调试脚本，用来先验证上传接口流程

## 2. 这个框架的设计思想

它不是把请求写死在测试函数里，而是把测试步骤拆成“数据”和“执行器”两部分：

- 用例内容放在 Excel
- 通用执行逻辑写在 `test_runner.py`
- 通过参数化实现“一套代码跑多条用例”

这就是典型的数据驱动自动化框架。

## 3. 这套框架已经具备的能力

- Excel 驱动用例执行
- 基于 `is_true` 开关控制是否执行
- 支持 `params / data / json / files`
- 支持 HTTP 响应断言
- 支持数据库断言
- 支持 JSON 提取
- 支持 SQL 提取
- 支持前后接口数据关联
- 支持 allure 报告
- 支持 pytest 日志落盘和控制台输出

## 4. 这套框架的执行流程

### 第一步：读取用例

`read_excel()` 读取 `Sheet1`，把第二行作为字段名，从第三行开始组装每条用例，并根据 `is_true` 过滤。

### 第二步：参数化执行

`@pytest.mark.parametrize("case", data)` 会让每一行 Excel 数据都生成一次独立测试执行。

### 第三步：模板渲染

执行用例前会先做：

```python
case = eval(Template(str(case)).render(all))
```

这样可以把 Excel 里的 `{{TOKEN}}` 替换成前置接口提取出来的真实值。

### 第四步：动态组装请求

框架根据 Excel 每列字段动态生成：

- 请求方法
- 路径
- 请求头
- params
- data
- json
- files

然后统一调用：

```python
requests.request(**request_data)
```

### 第五步：断言

断言分两类：

- 接口响应断言
- 数据库结果断言

### 第六步：提取

提取也分两类：

- `jsonExData`：从响应体提取
- `sqlExData`：从数据库结果提取

提取结果统一写入类属性 `all`，作为全局变量池供后续用例复用。

## 5. 这份示例数据在测什么

从 Excel 表头和数据内容可以看出，这个项目主要演示了三类接口流程：

- 登录成功并提取 `TOKEN`
- 未登录查询用户列表失败
- 带 `TOKEN` 查询用户列表成功
- 带 `TOKEN` 上传图片成功

因此，这套框架的教学重点不是业务复杂度，而是“接口自动化框架的最小闭环”。

## 6. 适合面试时怎么描述

可以概括成一句话：

> 这是一个基于 `pytest + requests + Excel + allure + pymysql + jinja2` 搭建的数据驱动接口自动化框架。测试数据存放在 Excel 中，执行层统一解析请求、做断言、做提取，并通过全局变量池完成接口之间的数据关联。

## 7. 当前实现的不足

这套代码适合教学，但工程化上还有明显提升空间：

- 大量使用 `eval()`，有安全风险
- 数据库连接重复创建，没有抽离公共连接层
- URL、数据库配置写死，不方便切环境
- 上传文件句柄没有统一管理
- `all` 作为类属性共享，扩展并发执行时要谨慎
- 用例仍依赖 Excel，规模变大后维护成本会升高

## 8. 一句话总结

这是一套“最小可运行”的接口自动化框架示例，已经覆盖了：

- 数据驱动
- 参数化执行
- 断言
- 提取
- 关联
- 报告

如果继续演进，下一步就应该补齐：

- 配置分层

- 更规范的测试报告与环境管理
