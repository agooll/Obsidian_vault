# WMS项目笔记 - 面试题汇总

> 来源：`测试开发/笔记/wms项目/assets/wms项目/5_web项目实战笔记(WMS项目).pdf`，按 PDF 中原顺序整理，并保留原优先级标记。

### 1. 能不能介绍一下你上一家公司？[2]

- 深圳某科技股份有限公司。
- 公司成立于某年，主营业务是定制化软件研发。
- 公司的信息化产品主要有商城、电商订单 OMS、仓库 WMS、运输 TMS 等系统。
- 公司总人数 200+，研发团队大约 50 人。
- 各项目之间有上下游关系：电商负责订单来源，OMS 处理订单数据，WMS 处理出入库数据，TMS 处理出库后的物流数据。

### 2. 可否展开介绍一下公司的盈利范围？[1]

- 售卖定制化软件：一次性付费，客户获得软件使用权，例如 WMS 可以单独销售，也可以和上下游产品打包销售。
- 软件维护服务：周期性付费，例如一年内的软件维护、升级、定制化开发。
- 软件使用过程中的抽佣：例如对商城订单或物流订单按比例抽取一定费用。

### 3. 研发团队构成？[3]

- 研发总数大约 50 人，分 4 个项目组，每个项目组大约 10+ 人。
- 我负责的是 WMS 项目，团队构成为：1 个产品 + 1 个 UI + 2 个前端开发 + 3 个后端开发 + 2 个测试。
- 如果负责的是商城项目，团队一般会更大，例如：1 个产品 + 2 个 UI + 3 个前端 + 5 个后端 + 3 个测试。
- 因为是定制化开发，不同客户在使用 WMS 时会有个性化需求，所以团队构成会动态调整。

### 4. 你的主要工作内容是什么？[3]

- 我主要负责 WMS 项目周迭代中的测试工作，包含功能测试和接口测试。
- 当上下游项目需要支援，或者出现数据缺失时，我也会提供支持。

### 5. 简单介绍一下 WMS 项目？[3]

1. WMS 是一个 B/S 架构的仓库管理系统。
2. 主要用于管理和优化仓库操作流程，提高仓库管理的效率和准确性。
3. 供采购、销售、客服、仓管等角色使用，覆盖采购、入库、存储、出库、承运等环节。
4. 采用前后端分离架构，前端技术栈为 `Vue + ElementUI`，后端技术栈为 `Spring Boot + JWT + Spring Security + Redis + MySQL`。

### 6. 项目中模块之间的关系是什么？[1]

- 一个系统通常由多个模块组成。
- 模块之间的关系分多种情况：
- 登录模块属于前置模块，要使用其他模块功能必须先登录。
- 角色管理、用户管理这类模块和权限相关，会约束不同用户能使用哪些功能。
- 客户管理和供应商管理相关性不强，更多是给不同业务角色使用。
- 入库和出库属于典型上下游业务关系，出库的数据必须建立在已入库的数据之上。

### 7. 你们公司的需求评审流程是怎么样的？[2]

1. 一般是周一早上产品经理先群发网页版 PRD 文档，让大家提前熟悉。
2. 大约一小时后，项目组的产品、UI、开发、测试一起开会。
3. 会议中通常由产品经理讲解需求，再由各方提问。
4. 一般一个上午会把评审走完。

### 8. 你在需求评审过程中，提出过哪些问题？[3]

- 最常见的是产品经理没有明确输入字段的长度和类型，这种问题需要在评审会上明确。
- 还有一些数据逻辑问题，不能只看原型图，要确认是模糊查询还是精确查询、数据唯一性、展示字段是否有意义、具体查表和排序逻辑。
- 还会关注异常处理逻辑，例如越权访问时页面和接口层面应该怎么报错。

### 9. 你们公司测试计划是怎么写的？[3]

1. 文档标题：`xx项目xx版本测试计划`。
2. 测试负责人：某某。
3. 测试策略：`UI + 功能 + 易用性 + 接口`。
4. 预期测试过程：新接口功能测试 + 冒烟测试 + 3 轮系统测试 + 接口层旧功能回归测试 + 验收测试。
5. 测试里程碑：例如周四早上提测，周五下午完成系统测试并开始验收，周五下班前结束验收。

### 10. 你们以前是如何管理测试用例的？[1]

- 我们使用企业版石墨文档管理测试用例。
- 用例以表格形式记录。
- 团队可以按版本号随时查看某个版本的测试用例内容和执行结果。
- 后续用例的维护和更新也都在石墨文档上完成，并同步给团队成员。

### 11. 你负责的项目有多少条测试用例？[3]

- 这个和项目大小有关。
- 我最近负责的 WMS 项目，每个迭代一般会新增 100 到 200 条测试用例。
- 高速迭代一年左右后，总用例数量大约有四五千条。

### 12. 你参与过开发评审会议吗？一般在会议上你关注哪些问题？[2]

- 每个迭代我都会参加开发评审会议。
- 一般由研发负责人发起，参与人员通常是前后端开发加测试，有时 UI 也会参加。
- 我主要关注新增了几个接口、旧功能有哪些修改项、数据库表结构和数据变更情况。

### 13. 测试用例评审过程中，你提出过哪些问题？[1]

- 从黑盒测试角度设计用例时，一个人的思维有限，多人评审能补足更多维度。
- 例如弱网状态下表单连续提交后结果会怎样，这种场景需要考虑。
- 例如登录过期时提交操作，会出现什么报错信息，能否跳转回登录页，这类异常也要关注。

### 14. 依赖登录的接口怎么测？[3]

- 先请求登录接口，获取 token。
- 再带着 token 去请求目标接口。

### 15. 有上下游关系的接口怎么测？[2]

- 先请求上游接口。
- 提取下游接口需要的数据。
- 再请求下游接口，并提交提取出来的数据。

### 16. 你们公司提测的标准是什么？[3]

- 公司规定开发在冒烟测试自测通过后，才可以正式提测。
- 冒烟测试用例由测试人员提供。

### 17. 回归测试你们是怎么做的？[3] 回归测试通过的标准是什么？[1]

- 我们主要通过接口回归来做旧功能回归。
- 因为高速迭代项目半年后用例数量就可能达到 2000+，如果从 UI 层面做大规模回归，会消耗大量人力和时间，所以更适合做接口层回归。
- 回归测试脚本执行成功、没有发现 bug，就视为回归测试通过。

### 18. 介绍一下你们公司的接口测试流程？[3]

- 接口测试只是测试的一个层级，具体流程和应用方向有关。
- 我之前公司的接口测试主要有两个应用方向：联调阶段的接口功能测试，以及旧功能的接口回归测试。
- 对于联调阶段的接口功能测试，流程上先从 Swagger 获取 API 文档，再用 JMeter 对每个新接口验证字段长度、字段类型、必填项、加解密等，同时验证接口对数据库的影响是否符合预期，例如数据逻辑和唯一性。
- 对于旧功能接口回归测试，流程上先执行上一个迭代形成的回归脚本，如果没有问题，再把本次新增的重要接口补进回归脚本，供下次回归继续使用。

### 19. 你们公司的验收测试是怎么做的？[1]

- 测试人员提供验收测试用例，由产品经理或者客户完成验收测试。
- 后来公司比较信任我，也会由我主导验收测试。
- 验收测试用例的设计重点是站在用户角度思考。
- 一般筛选 20% 到 30% 的核心正向用例即可，不必把字段验证这类细节全部带进去。
- 还要结合实际业务流程设计验收路径。
- 冒烟测试则一般只提供 5% 到 10% 的核心测试用例。

### 20. 介绍一下你们公司的测试流程？[3]

1. 每个迭代开始先收到产品经理发来的 PRD 文档，先做分析，再在需求评审中提问。
2. PRD 定稿后，测试组长编写测试计划。
3. 然后用 XMind 设计测试点，再整理成测试用例并发起评审。
4. 同期参加开发评审会议，并在开发期间协助接口联调、执行接口功能测试。
5. 提测前由测试准备好冒烟测试用例，开发搭环境并执行冒烟，通过后再正式提测。
6. 接下来进入系统测试，一般至少 3 轮，第一轮完整执行用例，后续多轮主要做 bug 回归。
7. 最后还会从接口层面做一轮旧功能回归测试。
8. 然后提交验收测试用例给产品经理，由其完成验收，测试这边输出完整测试报告。
9. 上线后也会做上线后的检查和持续跟进。

### 21. 接口测试你发现过哪些 bug？你如何定位的？[3][1]

- 常见 bug 类型包括：
- 边界问题：例如后端没有校验输入边界，或者和数据库字段限制不兼容。
- 异常问题：逻辑不完善导致极端异常没有被捕捉，严重时甚至可能让程序崩溃。
- 越权问题：页面只是隐藏了菜单，但接口层面实际仍可越权操作。
- 安全问题：例如入库完成状态的入库单本不允许修改，但接口层面仍可以成功修改。
- 并发问题：例如不同账号同时对最后一件物料做出库操作时，后端逻辑是否完善。
- 定位时一般会先结合接口响应、数据库数据和抓包结果一起判断，先区分是前端问题、后端问题，还是数据库逻辑问题。

### 22. 你们设计测试用例时的需求覆盖率是多少？测试用例执行率是多少？[1]

- 需求覆盖率和测试用例执行率都按 100% 去要求。
- 如果面试官继续深问“率”的定义，也可以反问分母是什么，再按具体口径回答。

### 23. 测试人员在工作中一般使用数据库做什么？[1]

- 场景很多，例如做功能测试时，需要结合数据库判断功能是否正确。
- 做分页测试时，可以直接往数据库里补数据。
- 出现 bug 时，也可以结合数据库做初步定位。
- 缺少测试数据时，可以直接去数据库改出自己想要的数据，而不是反复从页面点操作。
- 性能测试时，也常常需要通过数据库批量造数据。

### 24. 介绍一下你最近做的需求，做了多久，写测试用例用了多久，写了多少条，开发了多久，测试了多久，有什么测试点？[3]

- 我最近做的是 WMS 项目的入库模块测试。
- 整体按周迭代方式推进。
- 写测试用例一般会花 1 到 1.5 天，产出大约 150 条测试用例。
- 开发大约需要 2 天时间，测试总时长也大约是 2 天。
- 测试点主要围绕入库单的列表展示和增删改查来展开分析。

### 25. 有没有印象较深的 bug？如何定位的？后续开发是如何修复的？[3]

- 例如有一个 bug：作废状态的入库单，状态字段在前端页面上显示为空白。
- 定位过程是：
- 先查数据库，看到这条入库单对应状态字段值为 `Null`。
- 直接把状态值改为 `4` 后，前端页面能正常显示，因此初步排除后端渲染问题。
- 再用 F12 抓包发现，当把入库单状态改成“作废”时，前端并没有把对应值传给后端，因此最终定位为前端问题。
- 后续直接把 bug 提给前端开发修复。

### 26. 如果今晚马上上线了，但是开发才提测，这种情况你怎么解决？[3]

- 首先我实际没有遇到过这种情况。在团队里，规则比无效努力更重要，如果工作量超出预期，理论上应该在计划和开发方案阶段就暴露出来。
- 如果真的遇到这种情况，先专注解决问题，再思考如何避免同类问题重复发生。
- 我的处理思路是：
- 和开发团队密切合作，及时反馈 bug，快速响应。
- 做好优先级管理，先完成重要且紧急的测试工作。
- 多采用灰盒测试方法，结合程序设计逻辑高效验证，再配合黑盒测试。
- 能自动化完成的测试尽量自动化执行。
- 必要时及时和其他团队沟通，临时协调更多资源。

### 27. 马上要上线了，突然发现一个严重的 bug，而且这个 bug 短时间内解决不了，你打算怎么办？[3]

1. 先确认是否真的是 bug，排除数据和环境干扰。
2. 然后立即报告缺陷，必要时建议团队开紧急会议，评估这个 bug 的影响范围。
3. 一起讨论替代方案。如果继续上线，就要做好跟进验证和风险评估；如果影响太大，则建议延期上线。
4. 最后要复盘，分析为什么这么晚才发现这个 bug，以便优化团队工作流，避免同类问题再次发生。

### 28. 当一个面试题问得很细，比如技术细节或者业务细节，自己确实不清楚，该用什么思路来回答？[3]

1. 平时要尽量避免“喂饭式学习”，要多思考、多提问。
2. 如果面试时真遇到这类问题，可以按这个思路答：
3. 先如实说明自己没有直接遇到过这类问题，必要时说明当时是由谁负责的。
4. 再假设如果由自己处理，会怎么分析、怎么解决。
5. 最后尽量结合例子，把话题引导到自己更熟悉的领域上。
