# 接口自动化测试笔记 - 面试题汇总

> 来源：飞书云文档《接口自动化面试题》，按页面原顺序整理，并保留原优先级标记。

### 1. 你们的接口自动化测试是怎么做的？[3]

- 我们回归旧功能时发现耗时太久，所以开发了一个接口自动化框架。
- 这个框架是用 Python 写的，技术上用到了 `python + requests + pymysql + pytest + allure + 测试用例关键字驱动`。
- 框架的核心业务是我独立开发的，大概用了两周时间。
- 然后就开始补充测试用例，接口覆盖率达到 80%。
- 测试结束后我会查看 Allure 测试报告，分析报错，如果有 bug 就提交。

也可以分各种维度详细阐述。

从流程上来说：

- 先确认接口自动化测试的需求，主要包括目标项目和测试策略。一般情况下，测试框架搭建起来后在研项目都是可用的；策略上一般情况下是做接口回归测试。
- 然后是测试计划。在这个阶段需要协调可用资源，比如测试服务器、人手、技术选型等。我们一般情况下会用 Python 来实现。
- 然后就可以着手搭建环境、准备数据、编写测试框架，重点是搭建高复用的测试框架，这里会用到关键字驱动技术。
- 有了可用的框架，就可以着手选择和编写测试用例，并与框架联合调试。
- 最后就是执行和分析测试报告。

从团队分工来说：

- 当时我在项目组抽了两个人，一共 3 人着手开发接口自动化代码。我负责核心业务和工具，另外两个人负责组织测试数据、调试测试用例等工作。（如果是应届同学，可以说是自己独立搭建的框架。）
- 技术上选型用的是 `python + requests + pymysql + pytest + allure + 测试用例关键字驱动`。
- 我们主框架代码的开发时间大概用了两周，实现了 HTTP 和 JDBC 请求、HTTP 响应断言和数据库断言、JSON 提取和数据库提取等各种功能。

### 2. 接口自动化框架你是如何从 0 到 1 搭建的？[3]

1. 首先确认技术选型为 `python + requests + pymysql + pytest + allure + 测试用例关键字驱动`。
2. 确认核心业务流程为：测试用例解析 -> 全局变量引用 -> Allure 报告初始化 -> 发请求 -> 断言 -> 提取 -> 生成 Allure 测试报告。
3. 设计易扩展的项目结构，例如：
   - `testcases` 目录用于保存测试脚本。
   - `utils` 目录用于保存各种工具类或工具函数，如 Excel 用例读取、请求数据解析、Allure 报告初始化、请求、断言、提取器等。
   - `data` 目录用于保存测试用例文件。
   - `file` 目录用于保存待上传的文件。
   - `config` 目录用于保存项目的基础配置信息，如 `base_url`、数据库配置等。
   - 此外还有运行入口、胶水文件、日志配置文件、`log` 目录和 `report` 目录等。
4. 编写代码并调试。
5. 最后就是测试框架的使用过程。由于技术选型比较合理，代码维护工作量不大，且对后续执行人员的代码能力要求不高，小白也能上手，会写测试用例就能用。

### 3. 接口自动化中常用的库有哪些？[2]

- `requests`：负责 HTTP 请求。
- `pymysql`：负责 JDBC 请求。
- `openpyxl`、`jsonpath` 和 `jinja2`：负责测试用例的数据处理。
- `pytest` 及其各种插件：负责测试脚本的执行。
- `allure-pytest`：负责测试报告生成。
- `logging`：负责日志输出。

### 4. 你封装过哪些工具类？[2]

- 读取 Excel 测试用例文件的工具。
- HTTP 请求数据解析工具。
- Allure 测试报告初始化工具。
- 封装好的 HTTP 请求和 JDBC 请求。
- HTTP 响应断言和数据库断言。
- JSON 提取器和数据库提取器等。

### 5. 请求方式有哪些？每种请求方式都封装了吗？[1]

- 常见的请求方式有 `get`、`post`、`put`、`delete`。
- 我们观察 `requests` 的实现过程会发现，每种请求方式的实现都有底层的实现逻辑。例如 `requests.get()` 方法底层调用的是 `requests.request()` 方法，那么我就直接使用这个方法，方便统一传模板化的数据。
- 后面再次做封装是为了能够在 Allure 报告中展示这个核心步骤，所以封装了一层，并用装饰器 `@allure.step` 进行标记。

### 6. 接口之间的关联场景你们是怎么处理的？如果有动态的数据怎么处理？[3]

- 这种场景我们经常会遇到，例如 token 的使用、上下游业务之间的数据传递等。
- 处理方案一般情况下是先提取所需数据保存到全局变量中，再通过模板渲染的方式引用到下游接口请求中。

### 7. 接口自动化里的数据驱动或者关键字驱动是什么，怎么实现的？[3]

- 先说我对数据驱动的理解。数据驱动是一种依赖参数化技术的设计思想，规范化测试过程，让测试数据来决定测试结果，这样就能减少对测试脚本的直接维护。
- 关键字驱动是在数据驱动的基础上，额外增加一系列与步骤或操作有关的关键字数据。
- 我在框架中的实现是：
  - 先定义测试用例的模板数据和业务逻辑。
  - 再通过 `openpyxl` 封装从 Excel 文件中读数据的工具函数。
  - 最后在测试脚本中进行处理，如模板渲染和请求数据解析等。

### 8. 数据驱动的 Excel 文件都包含什么内容？测试用例文件中包含什么字段？[2]

- 首先是用例描述信息，如 id、模块、场景、标题。
- 然后是请求信息，如请求方式、路径、请求头、各种请求参数。
- 还有断言信息，如断言字段的 JSONPath 表达式、断言的预期结果、数据库断言的 SQL 语句和预期结果。
- 提取器信息，如用于从 HTTP 响应和数据库提取数据的相关字段。
- 以及分组执行信息。

### 9. 接口自动化测试用例编写的要点有哪些？[2]

- 一般用于从接口层进行回归测试，需要重复执行，需注意数据的初始化和清除操作。
- 对旧接口的覆盖率没必要达到 100%，需覆盖重要的和高频使用的接口。
- 挑选测试用例时，一般只选择正向的测试用例即可。

### 10. 如何读取 Excel 文件的每一行？[1]

- 读取 Excel 文件的操作我用的是 `openpyxl` 这个库实现的。
- 首先通过 `openpyxl.load_workbook()` 方法打开某个 Excel 文件。
- 然后通过工作表名这个 key 来选择对应的工作表。
- 再然后就可以通过工作表的 `iter_rows()` 方法得到行数据。由于得到的是一个可迭代对象，可以使用 `for` 循环遍历。

### 11. 你知道什么是 JSON 吗？在 Python 中如何处理 JSON 数据？[2]

- 知道，JSON 是一种数据表达的格式，由于结构清晰，使用很广泛。
- JSON 数据虽然格式清晰，但是在内存中的数据类型是字符串，不好处理，不能像字典一样轻松提取数据。
- 我所知道的能处理 JSON 数据的方法也有很多：
  - 如使用 `json` 这个库，通过 `loads` 方法可以把 JSON 字符串转为字典，通过 `dumps` 方法可以把字典转为 JSON 格式的字符串。
  - 我在框架中使用的是另一个办法，用到了 `eval()` 函数，它可以把字符串当成表达式来重新计算，也可以做到把 JSON 字符串转为字典。

### 12. 如何判定结果是符合预期还是不符合预期呢？[2]

- 这个比较简单，断言就可以做到。
- 我的断言方式是使用 Python 自带的 `assert` 关键字来实现的。

### 13. pytest 的作用是什么？[2]

- `pytest` 是一个单元测试框架。
- 主要作用是控制测试用例的执行过程。
- 它的功能很强大，通过自身功能和安装插件可以做到分组执行用例、控制执行顺序、失败重试、生成报告等等。

### 14. 在你的接口自动化测试框架中用到了哪些 pytest 插件？[2]

- `pytest-ordering`：控制执行顺序。
- `pytest-html`：生成测试报告。当然也可以不用，也可以用 `allure-pytest` 来生成更精美的 Allure 报告。
- `pytest-rerunfailures`：控制重复执行。

### 15. 为什么选择 pytest？比 unittest 好在哪里？[3]

- 作为第三方的 `pytest` 比 Python 自带的 `unittest` 功能更强大。
- 首先 `pytest` 的编写规则简单，兼容性更好，甚至可以直接执行 `unittest` 风格的用例，无需修改任何代码。
- `pytest` 可以自动发现项目中的测试脚本，`unittest` 需要手动编写测试套件。
- `pytest` 的断言非常灵活，`unittest` 的断言只提供了一些基本方法，如 `assertEqual`、`assertTrue` 等。
- `pytest` 有海量插件，`unittest` 不支持插件，功能相对固定。

### 16. pytest 测试用例的收集规则是怎样的？[2]

- `pytest` 有自动发现测试用例的功能，默认规则是：
  - 自动发现项目中 `test_` 开头的 py 文件。
  - 在文件中自动发现 `Test` 开头的测试类。
  - 在测试类中自动发现 `test_` 开头的测试函数。
- 当然也可以通过 `pytest.ini` 配置文件改变默认的发现规则。

### 17. 有 1000 条用例，我怎么选择其中 200 条用例来执行？[3]

- 可以实现的方法很多。
- 例如可以使用 markers 在 `pytest` 的配置文件中注册分组，再对测试用例做分组标记，执行时只需要通过 `-m` 参数就可以执行指定分组的用例了。
- 但是我更推荐既然是关键字驱动，就由关键字去处理。我会在测试用例文件中用一个字段来标记是否执行，只有需要执行的测试用例数据才会进行读取，这样控制更加方便。

### 18. 如何控制用例的执行顺序？[2]

- 可以实现的方法很多。
- 例如可以使用插件 `pytest-ordering` 的功能，先用装饰器 `@pytest.mark.run` 标记测试用例的执行顺序标号，再执行时就能自动识别顺序了。
- 但由于我是用 Excel 文件来管理测试用例的，我完全可以在文件中规划测试用例的编写顺序，也就是未来的执行顺序。

### 19. 有时候网络不好导致的测试用例执行失败，你是怎么处理的？[2]

- 可以通过 `pytest-rerunfailures` 插件的功能进行处理。
- 通过装饰器 `@pytest.mark.flaky(reruns=重试次数, reruns_delay=延时执行的秒数)` 来标记测试函数，就可以设置失败后的重试次数和延时重试的秒数。

### 20. 什么是装饰器？在你的框架中，你是怎么运用的？[3]

- 装饰器，我的理解就是修饰函数的函数，不改变原有函数代码的基础上，动态改变函数的功能。
- 在我的框架中，我对装饰器的使用一般是调用已有的即可，如 `@pytest.mark.run()`、`@pytest.mark.xfail`、`@pytest.mark.flaky()`、`@pytest.fixture` 等等。

### 21. 夹具的作用是什么？fixture 的 scope 几个作用域有什么区别？[2]

- 夹具本质上也是一个函数，它的作用类似 `setup` 和 `teardown`，允许我们在测试函数执行前后分别做一些初始化或销毁资源的动作。
- `fixture` 的几个作用域分别是：`function` 函数级、`class` 类级、`module` 模块级、`session` 会话级。区别是被 `@pytest.fixture()` 所标记的函数的执行时机不同。
- `function` 默认的函数级会在每个测试函数执行前后分别执行夹具所标记的函数对应的代码；同理，`session` 会话级在一次测试活动的前后总体只会执行一次对应的初始化和销毁资源的代码。

### 22. 胶水文件的作用？[2]

- 胶水文件是项目中的 `conftest.py` 文件，它比较特殊，和配置文件一样会在项目执行之前自动加载并运行。
- 一般情况下胶水文件可以把它理解为能写代码的 `pytest.ini` 文件。
- 经常用于定义测试项目中的夹具。

### 23. 什么是可迭代对象、迭代器、生成器？[2]

- 可迭代对象一般是能直接用于 `for` 循环遍历的数据对象，比如常见容器类型数据 `str`、`list`、`tuple`、`set`、`dict` 等都是可迭代对象，还有一些函数的返回值，如 `range()` 函数的返回值也是一个迭代的对象。
- 迭代器和可迭代对象很像，是可以被 `next()` 函数调用并不断返回下一个值的对象。但区别是可迭代对象一次性生成全部值，迭代器每次被 `next()` 函数调用只生成一个值，不会一次性全部生成。
- 生成器是一种特殊的迭代器，通过 `yield` 关键字来创建。生成器运行时每次遇到 `yield`，函数会暂停并保存当前运行信息，下次执行 `next()` 时从暂停位置继续运行。

### 24. 你在测试框架中是怎么实现 HTTP 请求、操作数据库、断言、提取器、日志的？[2]

HTTP 请求：

- HTTP 请求的实现，我用的是 `requests.request()` 方法。加载 Excel 文件中的数据，并把请求相关的数据解析出来传进去就可以了，后续封装到和请求相关的工具中。

操作数据库：

- 操作数据库的实现，我用的是 `pymysql` 这个库。把 JDBC 请求封装成工具函数，和 HTTP 请求同样放在请求相关的工具中，同时把数据库相关的配置信息封装到了配置文件中，方便修改。

断言：

- 断言我使用的是 Python 自带的 `assert` 关键字来实现的。
- 对于 HTTP 响应断言，我用到了相等和包含两种最常见的业务逻辑，而且随时可以扩展。
- 对于数据库断言的实现，调用之前封装过的 JDBC 请求来实现即可。最后断言的相关函数单独封装在断言工具中。

提取器：

- 有了请求的封装经验，提取器的实现过程会比较简单。JSON 提取器直接用 `jsonpath` 根据表达式提取响应结果中的部分字段就可以了，数据库提取调用 JDBC 请求就行。最后把提取器单独封装在工具中。

日志：

- 日志的实现主要是配置和输出两个部分，最明显的作用是观察程序的运行过程。
- 由于 `pytest` 和 `logging` 模块兼容性很好，日志的配置部分可以直接交给 `pytest.ini` 的参数，通过参数可以配置日志输出的位置、级别、格式化等内容。
- 日志的输出是分散到各个工具中的，覆盖了接口请求业务流程中的各个核心步骤，比如请求、响应、断言、提取等。

### 25. 接口测试完成后，如何清理垃圾数据？[3]

- 这是一个业务闭环问题。清理垃圾数据这个需求是由于接口自动化测试是需要重复执行的。
- 我会在执行测试用例的过程中记录产生垃圾数据的位置，并且把清理垃圾数据对应的 SQL 语句放在 `conftest.py` 的夹具中进行处理，夹具的作用域设置为 `session` 级别即可。测试执行到最后一条用例结束时就会自动清理冗余数据了。

### 26. 你认为什么样的项目适合做接口自动化？[2]

- 首先自动化是一种可以提高效率的手段，那么我们考虑一个项目是否适合做自动化，衡量的其实是成本和收益的关系。
- 如果实现自动化后，需要不断投入人力和时间成本去维护代码，或者接口代码经常改动，让测试脚本没办法重复使用，这种类型的项目不太适合。
- 在项目稳定的情况下，从长远来看，大多数项目都是适合做自动化测试的。

### 27. 你们自动化测试的执行时机和节奏是怎样的？是开发部署好环境后就开始执行吗？[1]

- 我们之前做接口自动化的策略是从接口层对旧功能做回归测试，一般是上线前会跑一次。
- 因为执行效率较高，每个迭代版本都会执行。

### 28. 你们的接口自动化测试用例跑一次需要多久？有没有想过如何优化？[2]

- 我们一般情况下跑一次自动化测试的用例数大概是三四百条，大概需要两三分钟可以跑完。
- 个人认为这个时间对于接口测试来说算是比较合理的，也可以接受，暂时可以不去做优化。但是我也曾考虑过这个问题，如果后续需要优化，可以采用分布式执行的办法，刚好 `pytest-xdist` 插件就支持这个功能。

### 29. 你们接口自动化的用例数、覆盖率、通过率数据是多少？[3]

- 上个项目前后台的接口总数大概近 400 个。（根据实际情况出发，视项目大小而定，200 到 400 这个区间都是合理的。）
- 我们做接口自动化的覆盖率达到了 80%，覆盖的回归测试用例 300 多条。
  - 还有一部分接口没有覆盖的原因是：某些功能对应的接口使用率较低，还有部分接口涉及第三方付费服务，例如查快递物流信息、发短信验证码等。
- 通过率基本都是 100%，毕竟都是旧功能了。

### 30. 接口自动化测试报告都有哪些内容？[1]

- 我们生成测试报告用的是 Allure。
- 进入 Allure 报告默认看到的是总览数据，包含用例数、通过率、场景信息等。
- 还有一些用得不多，比如测试套、图表概览、时间性能展示这类功能。
- 主要看的是功能页面标签下所有测试用例的描述信息和执行过程，也可以在这里看到报错信息。

### 31. 一般 Allure 测试报告中会出现什么样的报错信息？你在自动化接口测试过程中发现过哪些 bug？[3]

接口自动化测试过程中的报错一般都是断言失败，原因一般可以分几种情况：

- 项目代码或数据问题：开发在未告知测试的情况下修改了旧接口的代码，接口文档也没有更新，或者对数据库结构做了调整，可能会导致测试脚本执行报错。
- 第三方接口问题：接口不稳定，偷偷更新了。
- 测试环境的网络条件不稳定，可能出现超时错误。
- 多人开发协作过程中代码管理流程不完善，可能出现有人用旧代码覆盖新代码的情况。

### 32. 接口自动化如果涉及验证码，怎么实现？[3]

之前有遇到过需要处理验证码的情况，我会分为上线后和上线前两种情况去考虑。

对于线上环境，最好不要留后门：

- 如果是图片验证码，我会优先考虑 OCR 去识别图片上的字符，其实识别率不高，还不到 50%。
- 如果是短信验证码，我会到验证码的存储位置去读，比如数据库读取。

对于测试环境，相对比较简单：

- 可以让开发直接关闭验证码或者设置万能验证码。
- 也可以将部分账号添加到白名单来跳过验证。

### 33. 一个不可逆的操作，如删除订单后，如何确认操作是否成功？[1]

- 对于 JSON 格式的响应结果，可以直接通过 HTTP 响应断言来进行判断。
- 如果担心不够严谨，还可以在 HTTP 响应断言的基础上再同时使用数据库断言。

### 34. 一个 HTTP 请求的响应结果每次返回的数据都不一样，怎么去断言？[1]

- 优先从业务逻辑的方向去进行思考，要知道为什么每次响应都不一样，例如是加密方式导致的，还是时间戳导致的。找到原因后，我们可以维护代码去增加对应的功能。
- 当然也可以考虑直接从数据库进行断言。

### 35. 你们接口自动化有没有做线上巡检？怎么做的？有没有发现过什么问题？[2]

- 有做线上巡检。
- 做线上巡检意味着配置的 IP 和数据库都是生产环境，策略上要比较谨慎，原则上不能改动线上的数据，所以线上巡检只覆盖部分查询类接口。
- 我之前的公司线上巡检没发现过什么问题，毕竟在测试环境测试得比较充分。

### 36. 生产环境的接口都是非对称加密的，这种情况你有什么解决方案吗？[2]

- 这里有两个疑问点：1 是生产环境，2 是非对称加密。
- 先说 1，针对生产环境做线上巡检是很严格的，限制诸多，更推荐在测试环境解决问题。
- 再说 2，在测试环境处理加密的方法大致也可以分为绕过和处理两类：
  - 如果是绕过，那么关闭加解密函数即可。
  - 如果要处理，那么就需要增加加解密的业务环节，并在 Excel 文档中增加对应的字段，用以控制是否需要加解密，以及具体使用哪一种加解密方法，在自动化测试框架中进行判断并调用对应的工具就行了。

### 37. 接口的场景你们在框架里面是怎么实现的？一个 case 如果对应多个接口，怎么做断言？[3]

删除用户场景：1 登录 -> 2 新增用户 -> 3 修改用户状态 -> 4 新增角色 -> 5 新增权限 -> 6 删除用户。（涉及 6 个接口）

- 我们较少处理这种情况，一般情况下一条用例对应一个接口的请求。
- 如果需要一条用例对应多个接口的请求，形成了一个场景，那么我们的测试框架也是支持的。Allure 测试报告中有个字段叫作 `feature`，原意是功能，我们 Excel 测试用例中也有这个字段，用于标识功能模块。我们可以转念想想，如果用它来标识大的功能场景也是可以的。
- 我可以把多个接口的请求按照顺序要求设置为同一个场景，那么根据场景的维度去查看结果就可以了。
- 断言的方式是每个请求单独拥有自己的断言，一个场景中所有接口请求都断言成功则表示这个场景测试通过。

### 38. 你们编写接口自动化测试框架时有没有什么难点？有什么可以优化的地方？[3]

- 先说难点，当时整理业务逻辑的过程比较顺利，倒是有个卡点。由于接口发起请求之前需要引用全局变量中的数据，例如 token，但是这个引用过程又需要写在 Excel 文件中。后来查阅了资料，发现了 `jinja2` 这个库的模板渲染功能可以解决这个技术问题，算是印象比较深刻的。
- 我们的接口自动化测试框架目前是可以满足需求的，也考虑过优化方向。比如可以多增加一些支持的加解密方式，可以增加单接口请求的多重断言功能，可以多增加几种提取数据的方式，例如正则方式提取等等。
