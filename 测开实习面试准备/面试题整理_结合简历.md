## 一、WMS项目

### 1. 介绍一下你的 WMS 项目

**答案：**

- WMS 是一个 B/S 架构的仓库管理系统。
- 主要用于管理和优化仓库操作流程，提高仓库管理的效率和准确性。
- 供采购、销售、客服、仓管等角色使用，覆盖采购、入库、存储、出库、承运等环节。
- 采用前后端分离架构，前端技术栈为 `Vue + ElementUI`，后端技术栈为 `Spring Boot + JWT + Spring Security + Redis + MySQL`。

### 2. 你在 WMS 项目里主要负责什么

**答案：**

- 我主要负责 WMS 项目周迭代中的测试工作，包含功能测试和接口测试。
- 当上下游项目需要支援，或者出现数据缺失时，我也会提供支持。

### 3. 你们团队研发构成是怎样的

**答案：**

- 研发总数大约 50 人，分 4 个项目组，每个项目组大约 10+ 人。
- 我负责的是 WMS 项目，团队构成为：1 个产品 + 1 个 UI + 2 个前端开发 + 3 个后端开发 + 2 个测试。
- 因为是定制化开发，不同客户在使用 WMS 时会有个性化需求，所以团队构成会动态调整。

### 4. 你测了哪些模块

**答案：**

- 我主要围绕登录鉴权、入库、出库、库存查询、盘点、权限管理这些模块开展测试。
- 这些模块里，登录属于前置模块，权限管理会影响不同角色的使用范围，入库和出库又属于典型上下游关系。

### 5. 你在需求评审过程中会提哪些问题

**答案：**

- 最常见的是产品经理没有明确输入字段的长度和类型，这种问题需要在评审会上明确。
- 还有一些数据逻辑问题，不能只看原型图，要确认是模糊查询还是精确查询、数据唯一性、展示字段是否有意义、具体查表和排序逻辑。
- 还会关注异常处理逻辑，例如越权访问时页面和接口层面应该怎么报错。

### 6. 你是怎么拆测试点的

**答案：**

- 我会先根据需求文档和业务流程拆主链路，再补异常、边界和权限场景。
- 在需求评审时我会重点看字段长度和类型、数据唯一性、查询逻辑、排序逻辑、异常处理逻辑，以及越权访问时页面和接口层面的报错。
- 在用例评审时，也会补充弱网、登录过期、连续提交这类异常场景。

### 7. 你们公司测试计划是怎么写的

**答案：**

1. 文档标题：`xx项目xx版本测试计划`。
2. 测试负责人：某某。
3. 测试策略：`UI + 功能 + 易用性 + 接口`。
4. 预期测试过程：新接口功能测试 + 冒烟测试 + 3 轮系统测试 + 接口层旧功能回归测试 + 验收测试。
5. 测试里程碑：例如周四早上提测，周五下午完成系统测试并开始验收，周五下班前结束验收。

### 8. 你们公司的测试流程是怎样的

**答案：**

1. 每个迭代开始先收到产品经理发来的 PRD 文档，先做分析，再在需求评审中提问。
2. PRD 定稿后，测试组长编写测试计划。
3. 然后用 XMind 设计测试点，再整理成测试用例并发起评审。
4. 同期参加开发评审会议，并在开发期间协助接口联调、执行接口功能测试。
5. 提测前由测试准备好冒烟测试用例，开发搭环境并执行冒烟，通过后再正式提测。
6. 接下来进入系统测试，一般至少 3 轮，第一轮完整执行用例，后续多轮主要做 bug 回归。
7. 最后还会从接口层面做一轮旧功能回归测试。
8. 然后提交验收测试用例给产品经理，由其完成验收，测试这边输出完整测试报告。
9. 上线后也会做上线后的检查和持续跟进。

### 12. 你们公司提测的标准是什么

**答案：**

- 公司规定开发在冒烟测试自测通过后，才可以正式提测。
- 冒烟测试用例由测试人员提供。

### 13. 回归测试你们是怎么做的

**答案：**

- 我们主要通过接口回归来做旧功能回归。
- 因为高速迭代项目半年后用例数量就可能达到 2000+，如果从 UI 层面做大规模回归，会消耗大量人力和时间，所以更适合做接口层回归。
- 回归测试脚本执行成功、没有发现 bug，就视为回归测试通过。

### 14. 你们公司的接口测试流程是怎样的

**答案：**

- 接口测试主要有两个应用方向：联调阶段的接口功能测试，以及旧功能的接口回归测试。
- 对于联调阶段的接口功能测试，流程上先从 Swagger 获取 API 文档，再用 JMeter 对每个新接口验证字段长度、字段类型、必填项、加解密等。
- 同时还要验证接口对数据库的影响是否符合预期，例如数据逻辑和唯一性。
- 对于旧功能接口回归测试，流程上先执行上一个迭代形成的回归脚本，如果没有问题，再把本次新增的重要接口补进回归脚本，供下次回归继续使用。

### 15. 介绍一下你最近做的需求，做了多久，写测试用例用了多久，写了多少条，开发多久，测试多久，有什么测试点

**答案：**

- 我最近做的是 WMS 项目的入库模块测试。
- 整体按周迭代方式推进。
- 写测试用例一般会花 1 到 1.5 天，产出大约 150 条测试用例。
- 开发大约需要 2 天时间，测试总时长也大约是 2 天。
- 测试点主要围绕入库单的列表展示和增删改查来展开分析。

### 16. 接口测试你发现过哪些 bug？你如何定位

**答案：**

- 常见 bug 类型包括边界问题、异常问题、越权问题、安全问题和并发问题。
- 例如：
  - 后端没有校验输入边界，或者和数据库字段限制不兼容。
  - 逻辑不完善导致极端异常没有被捕捉。
  - 页面只是隐藏了菜单，但接口层面实际仍可越权操作。
  - 入库完成状态的入库单本不允许修改，但接口层面仍可以成功修改。
  - 不同账号同时对最后一件物料做出库操作时，后端逻辑是否完善。
- 定位时一般会先结合接口响应、数据库数据和抓包结果一起判断，先区分是前端问题、后端问题，还是数据库逻辑问题。

### 17. 讲一个你印象比较深的 bug

**答案：**

- 例如有一个 bug：作废状态的入库单，状态字段在前端页面上显示为空白。
- 定位过程是：
  - 先查数据库，看到这条入库单对应状态字段值为 `Null`。
  - 直接把状态值改为 `4` 后，前端页面能正常显示，因此初步排除后端渲染问题。
  - 再用 F12 抓包发现，当把入库单状态改成“作废”时，前端并没有把对应值传给后端，因此最终定位为前端问题。
- 后续直接把 bug 提给前端开发修复。

### 18. 如果今晚马上上线了，但是开发才提测，这种情况你怎么解决

**答案：**

- 首先我实际没有遇到过这种情况。在团队里，规则比无效努力更重要，如果工作量超出预期，理论上应该在计划和开发方案阶段就暴露出来。
- 如果真的遇到这种情况，先专注解决问题，再思考如何避免同类问题重复发生。
- 我的处理思路是：
  - 和开发团队密切合作，及时反馈 bug，快速响应。
  - 做好优先级管理，先完成重要且紧急的测试工作。
  - 多采用灰盒测试方法，结合程序设计逻辑高效验证，再配合黑盒测试。
  - 能自动化完成的测试尽量自动化执行。
  - 必要时及时和其他团队沟通，临时协调更多资源。

### 19. 马上要上线了，突然发现一个严重 bug，短时间内解决不了，你怎么办

**答案：**

1. 先确认是否真的是 bug，排除数据和环境干扰。
2. 然后立即报告缺陷，必要时建议团队开紧急会议，评估这个 bug 的影响范围。
3. 一起讨论替代方案。如果继续上线，就要做好跟进验证和风险评估；如果影响太大，则建议延期上线。
4. 最后要复盘，分析为什么这么晚才发现这个 bug，以便优化团队工作流，避免同类问题再次发生。

### 20. 当面试题问得很细，自己确实不清楚，该怎么回答

**答案：**

1. 平时要尽量避免“喂饭式学习”，要多思考、多提问。

### 21. 测试人员在项目里为什么要会查数据库

**答案：**

- 做功能测试时，需要结合数据库判断功能是否正确。
- 做分页测试时，可以直接往数据库里补数据。

## 二、接口测试 / 接口自动化

### 1. 你们的接口自动化测试是怎么做的

**答案：**

- 我们回归旧功能时发现耗时太久，所以开发了一个接口自动化框架。
- 这个框架是用 Python 写的，技术上用到了 `python + requests + pymysql + pytest + allure + 测试用例关键字驱动`。
- 框架的核心业务是我独立开发的，大概用了两周时间。
- 然后就开始补充测试用例，接口覆盖率达到 80%。
- 测试结束后我会查看 Allure 测试报告，分析报错，如果有 bug 就提交。

### 2. 接口自动化框架你是如何从 0 到 1 搭建的

**答案：**

1. 首先确认技术选型为 `python + requests + pymysql + pytest + allure + 测试用例关键字驱动`。
2. 确认核心业务流程为：测试用例解析 -> 全局变量引用 -> Allure 报告初始化 -> 发请求 -> 断言 -> 提取 -> 生成 Allure 测试报告。
3. 设计易扩展的项目结构，例如 `testcases`、`utils`、`data`、`file`、`config` 等目录。
4. 编写代码并调试。
5. 最后就是使用过程。由于技术选型比较合理，代码维护工作量不大，后续执行人员只要会写测试用例就能用。

### 3. 接口自动化中常用的库有哪些

**答案：**

- `requests`：负责 HTTP 请求。
- `pymysql`：负责 JDBC 请求。
- `openpyxl`、`jsonpath` 和 `jinja2`：负责测试用例的数据处理。
- `pytest` 及其插件：负责测试脚本执行。
- `allure-pytest`：负责测试报告生成。
- `logging`：负责日志输出。

### 4. 你封装过哪些工具类

**答案：**

- 读取 Excel 测试用例文件的工具。
- HTTP 请求数据解析工具。
- Allure 测试报告初始化工具。
- 封装好的 HTTP 请求和 JDBC 请求。
- HTTP 响应断言和数据库断言。
- JSON 提取器和数据库提取器等。

### 5. 接口之间的关联场景怎么处理

**答案：**

- 这种场景经常会遇到，比如 token 的使用、上下游业务之间的数据传递。
- 处理方案一般是先提取所需数据保存到全局变量中，再通过模板渲染的方式引用到下游接口请求中。

### 6. 数据驱动和关键字驱动是什么，怎么实现

**答案：**

- 数据驱动是一种依赖参数化技术的设计思想，让测试数据来决定测试结果，减少对测试脚本的直接维护。
- 关键字驱动是在数据驱动基础上，额外增加一系列与步骤或操作有关的关键字数据。
- 我的实现方式是：
  - 先定义测试用例模板数据和业务逻辑。
  - 再通过 `openpyxl` 封装从 Excel 文件中读数据的工具函数。
  - 最后在测试脚本中进行处理，如模板渲染和请求数据解析等。

### 7. 数据驱动的 Excel 文件一般包含什么字段

**答案：**

- 用例描述信息，如 id、模块、场景、标题。
- 请求信息，如请求方式、路径、请求头、各种请求参数。
- 断言信息，如断言字段的 JSONPath 表达式、断言预期结果、数据库断言 SQL 和预期结果。
- 提取器信息，如从 HTTP 响应和数据库提取数据的相关字段。
- 以及分组执行信息。

### 8. 如何读取 Excel 文件的每一行

**答案：**

- 读取 Excel 文件我用的是 `openpyxl`。
- 先通过 `openpyxl.load_workbook()` 打开 Excel。
- 再通过工作表名选择对应工作表。
- 然后通过工作表的 `iter_rows()` 得到行数据，再用 `for` 循环遍历。

### 9. JSON 在 Python 中如何处理

**答案：**

- JSON 是一种数据表达格式，由于结构清晰，使用很广泛。
- 可以使用 `json` 这个库，通过 `loads` 把 JSON 字符串转成字典，通过 `dumps` 把字典转成 JSON 格式字符串。
- 我在框架里也用过 `eval()` 去把字符串重新计算成字典。

### 10. 你怎么做断言

**答案：**

- 断言我使用的是 Python 自带的 `assert` 关键字来实现。
- 对于 HTTP 响应断言，我用到了相等和包含两种最常见的业务逻辑。
- 如果担心不够严谨，还可以在 HTTP 响应断言基础上再同时使用数据库断言。

### 11. 为什么选择 pytest，而不是 unittest

**答案：**

- 作为第三方的 `pytest` 比 Python 自带的 `unittest` 功能更强大。
- `pytest` 编写规则简单，兼容性更好，甚至可以直接执行 `unittest` 风格的用例。
- `pytest` 可以自动发现项目中的测试脚本，`unittest` 需要手动编写测试套件。
- `pytest` 的断言非常灵活。
- `pytest` 有海量插件，方便做分组执行、执行顺序、失败重试、报告生成等。

### 12. fixture 的作用是什么

**答案：**

- 夹具本质上也是一个函数，它的作用类似 `setup` 和 `teardown`，允许我们在测试函数执行前后分别做一些初始化或销毁资源的动作。
- `fixture` 的几个作用域分别是：`function`、`class`、`module`、`session`。
- 区别在于执行时机不同，比如 `function` 是每个测试函数前后执行，`session` 是整个测试活动前后只执行一次。

### 13. 有 1000 条用例，你怎么只选 200 条来执行

**答案：**

- 可以实现的方法很多。
- 例如可以使用 markers 在 `pytest` 的配置文件中注册分组，再对测试用例做分组标记，执行时只需要通过 `-m` 参数就可以执行指定分组的用例。
- 但是我更推荐既然是关键字驱动，就由关键字去处理。我会在测试用例文件中用一个字段来标记是否执行，只有需要执行的测试用例数据才会进行读取，这样控制更加方便。

### 14. 你在测试框架中是怎么实现 HTTP 请求、数据库、断言、提取器、日志的

**答案：**

**HTTP 请求：**

- HTTP 请求的实现，我用的是 `requests.request()` 方法。
- 加载 Excel 文件中的数据，并把请求相关的数据解析出来传进去，后续封装到请求工具中。

**操作数据库：**

- 我用的是 `pymysql` 这个库。
- 把 JDBC 请求封装成工具函数，和 HTTP 请求同样放在请求相关工具中，同时把数据库相关配置放到配置文件中，方便修改。

**断言：**

- 断言我使用的是 Python 自带的 `assert`。
- HTTP 响应断言支持相等和包含。
- 数据库断言就是调用之前封装过的 JDBC 请求来实现。

**提取器：**

- JSON 提取器直接用 `jsonpath` 根据表达式提取响应结果中的字段。
- 数据库提取调用 JDBC 请求就行。

**日志：**

- 日志的实现主要是配置和输出两个部分。
- 由于 `pytest` 和 `logging` 模块兼容性很好，日志配置部分可以直接交给 `pytest.ini` 参数。
- 日志输出会分散到请求、响应、断言、提取等核心步骤中。

### 15. 接口自动化完成后如何清理垃圾数据

**答案：**

- 这是一个业务闭环问题。因为接口自动化测试是需要重复执行的。
- 我会在执行测试用例过程中记录产生垃圾数据的位置，并把清理垃圾数据对应的 SQL 语句放在 `conftest.py` 的夹具中进行处理。
- 夹具作用域设置为 `session` 级别即可，测试执行完后就自动清理冗余数据。

### 16. 你们接口自动化的用例数、覆盖率、通过率数据是多少

**答案：**

- 上个项目前后台的接口总数大概近 400 个。
- 我们做接口自动化的覆盖率达到了 80%，覆盖的回归测试用例 300 多条。
- 还有一部分接口没有覆盖的原因是：某些功能对应的接口使用率较低，还有部分接口涉及第三方付费服务，例如查快递物流信息、发短信验证码等。
- 通过率基本都是 100%，毕竟都是旧功能了。

### 17. Allure 测试报告中一般会出现什么报错信息？你在自动化测试里发现过哪些 bug

**答案：**

- 接口自动化测试过程中的报错一般都是断言失败，原因一般可以分几种情况：
  - 项目代码或数据问题：开发在未告知测试的情况下修改了旧接口的代码，接口文档也没有更新，或者对数据库结构做了调整，可能会导致测试脚本执行报错。
  - 第三方接口问题：接口不稳定，偷偷更新了。
  - 测试环境网络条件不稳定，可能出现超时错误。
  - 多人开发协作过程中代码管理流程不完善，可能出现有人用旧代码覆盖新代码的情况。

### 18. 什么样的项目适合做接口自动化

**答案：**

- 自动化是一种提高效率的手段，衡量一个项目是否适合做自动化，核心是看成本和收益。
- 如果实现自动化后，需要不断投入很多人力和时间去维护，或者接口代码经常改动导致测试脚本没办法重复使用，这种类型的项目不太适合。
- 在项目稳定的情况下，从长远来看，大多数项目都是适合做自动化测试的。

### 19. 接口自动化如果涉及验证码，怎么实现

**答案：**

- 对于线上环境，最好不要留后门。
- 如果是图片验证码，我会优先考虑 OCR 去识别图片上的字符，但识别率不一定高。
- 如果是短信验证码，我会到验证码的存储位置去读，比如数据库读取。
- 对于测试环境，相对简单：
  - 可以让开发直接关闭验证码或者设置万能验证码。
  - 也可以将部分账号添加到白名单来跳过验证。

### 20. 一个 HTTP 请求的响应结果每次都不一样，怎么断言

**答案：**

- 优先从业务逻辑方向去思考，要知道为什么每次响应都不一样，例如是加密方式导致的，还是时间戳导致的。
- 找到原因后，可以维护代码去增加对应功能。
- 当然也可以考虑直接从数据库进行断言。

### 21. 一个 case 如果对应多个接口，怎么做场景断言

**答案：**

- 一般情况下是一条用例对应一个接口请求。
- 如果需要一条用例对应多个接口请求，形成一个场景，那么框架也是支持的。
- 我可以把多个接口的请求按照顺序要求设置为同一个场景，然后根据场景维度去查看结果。
- 断言方式是每个请求单独拥有自己的断言，一个场景中所有接口请求都断言成功，就表示这个场景测试通过。

### 22. 你们编写接口自动化框架时有没有什么难点

**答案：**

- 当时整理业务逻辑整体比较顺利，但有个卡点是：接口发起请求之前需要引用全局变量中的数据，例如 token，而且这个引用过程又需要写在 Excel 文件中。
- 后来查阅资料，发现了 `jinja2` 这个库的模板渲染功能可以解决这个技术问题，这一点印象比较深。

## 三、接口测试基础

### 1. 什么是接口

**答案：**

- 接口就是后端开放给外部使用的函数。

### 2. 接口测试怎么定位 bug

**答案：**

- 接口测试是一种绕过前端、直接验证后端逻辑的测试方法。
- 所以当接口测试通过的时候，说明后端接口大概率正常，这时候很大原因是前端问题。
- 如果后端接口测试不通过的情况下，并不能直接说问题一定出在前端或者后端，还要结合请求、响应和数据库一起判断。

### 3. GET 和 POST 有什么区别

**答案：**

- **安全性：**
  - GET 的参数直接显示在 URL 中，是明文传输。
  - POST 的参数一般放在 body 里面。
- **幂等性：**
  - GET 方法是幂等的，多次请求不会对服务器产生副作用。
  - POST 方法一般是非幂等的，多次执行相同 POST 可能会改变资源状态。
- **数据长度限制：**
  - GET 因为参数在 URL 中，所以输入长度有限制。
  - POST 因为数据放在 body 中，能承载的数据更多。
- **缓存：**
  - GET 一般有缓存。
  - POST 一般没有缓存。

### 4. 常见的状态码有哪些

**答案：**

- 1XX：继续
- 2XX：响应成功
- 3XX：重定向
- 4XX：客户端向服务器发送请求出问题
- 5XX：服务端向客户端发送数据出问题

### 5. HTTP 接口有哪些类型

**答案：**

- GET
- POST
- PUT
- DELETE

### 6. HTTP 和 HTTPS 的区别

**答案：**

- HTTP 是明文传输。
- HTTPS 使用 SSL/TLS 加密。
- HTTP 默认 80 端口，HTTPS 默认 443 端口。
- HTTPS 需要数字证书来验证服务器身份。
- HTTPS 因为有加密解密操作，性能会略低于 HTTP，但安全性更高。

### 7. 接口测试流程是怎样的

**答案：**

1. 需求分析、需求评审。
2. 编写测试计划，这个阶段可以提前引入接口测试，拿着 API 文档根据 URL、参数、请求方式等去设计接口测试用例，做简单的通过性测试。
3. 系统测试阶段，在开发做完冒烟之后，可以进行接口功能测试。接口测试绕过前端，直接验证后端逻辑，容易测出一些隐蔽的安全隐患。
4. 测试回归阶段，几轮 bug 回归以及旧功能回归，到最后可以使用接口自动化脚本去代替人工点点点。
5. 验收和报告，把执行结果和发现的缺陷汇总成测试报告。

### 8. 如何使用 JMeter 做接口测试

**答案：**

- 先根据接口文档配置请求方式、URL、请求头和参数。
- 如果有登录态或者上下游依赖，就用 JSON 提取器或正则表达式提取器把 token、订单号这类数据提取出来，再传给后续接口。
- 然后通过响应断言、JSON 断言、状态码断言去校验结果。
- 如果需要批量数据，就配合 CSV 参数化。

### 9. 如何写接口测试用例

**答案：**

- 接口测试用例本质上还是围绕输入、处理、输出和数据库结果去设计。
- 重点要覆盖正常场景、异常场景、边界值场景、权限场景和安全场景。
- 对于联调阶段的新接口，还要重点验证字段长度、字段类型、必填项、加解密、数据逻辑和唯一性。

## 四、MySQL / SQL

### 1. 数据库中有哪些常见的数据类型

**答案：**

- int
- char
- varchar
- datetime

### 2. delete / truncate / drop 的区别

**答案：**

- `delete` 删除数据的同时，不会重置自增，也不会破坏表结构。
- `truncate` 删除数据的同时，会重置自增，不会破坏表结构，而且只能 `truncate 表名`，不能加 where。
- `drop` 删除数据的同时，会把表结构也删除掉。

### 3. where 和 having 的区别

**答案：**

- where 是直接在表里面过滤。
- having 是分组后过滤。

### 4. SQL 语句综合书写顺序

**答案：**

1. select
2. from
3. where
4. group by
5. having
6. order by
7. limit

### 5. 内连接、左连接和右连接的区别

**答案：**

- 内连接是只显示两个表中相同的字段。
- 左连接是在左表连接右表时，右表没有对应数据的字段会显示为 `NULL`，但保留左表数据。
- 右连接反过来，保留右表数据。

### 6. 你什么时候会用到存储过程

**答案：**

- 在需要大量插入数据到数据库的时候，可以使用存储过程。
- 因为它相当于一个 SQL 函数，可以在函数内部定义相应逻辑，最后直接用 `CALL` 调用。

### 7. 你用过 Redis，有没有发现过什么问题

**答案：**

- 我遇到过缓存雪崩这类问题。
- 缓存雪崩是指 Redis 中大量缓存数据在同一时间失效，或者 Redis 服务异常不可用，导致原本应该访问缓存的请求全部打到 MySQL 数据库上。
- 如果请求量很大，MySQL 承受不了压力，就可能出现响应变慢甚至数据库崩溃。
- 解决思路可以是：给缓存设置随机过期时间、使用多级缓存，或者在后端增加限流、降级、请求队列等机制。

### 8. 使用数据库时，遇到过哪些 bug

**答案：**

- **数据类型设置不合理：**
  - 比如字段长度设计太短，导致真实业务数据被截断。
- **物理删除导致业务逻辑缺陷：**
  - 例如员工离职直接 `delete`，后续系统无法识别二次入职。
  - 商品下架如果直接物理删除，后续再次上架就要重新录入全部信息。
- 所以这类场景更适合使用逻辑删除，而不是物理删除。

### 9. 你在项目里怎么做数据库校验

**答案：**

- 调用接口后，我会通过 SQL 去查对应表的数据结果。
- 重点看接口操作有没有正确落库，关键字段是否符合预期。
- 如果是库存相关场景，还会重点校验库存增加、扣减、修正后的结果是否正确，避免出现接口成功但业务数据异常。

## 六、性能测试

### 1. 什么是吞吐量

**答案：**

- 吞吐量表示服务器的处理能力。
- 吞吐量高就代表服务器处理能力强。

### 2. QPS 和 TPS 有什么区别

**答案：**

- QPS 是服务器每秒的请求数。
- TPS 是服务器每秒的事务数。
- 一个事务可能包含一个或多个请求。

### 3. 吞吐量和平均响应时间是什么关系

**答案：**

- 它们通常是呈现反比关系的。
- 吞吐量高的话，平均响应时间一般就更低。

### 4. 性能测试的测试环境有什么要求

**答案：**

- 性能测试的测试环境尽量保证和生产环境一样。
- 这样能保证性能测试结果更有参考价值。

### 5. 性能测试你是怎么做的

**答案：**

- 我会先确定压测目标和场景，比如登录、入库、出库、库存查询这些核心操作。
- 然后准备好参数化数据、token、多用户数据和基础环境。
- 使用 JMeter 模拟并发用户执行这些核心操作。
- 压测过程中重点关注响应时间、吞吐量、错误率等指标。
- 同时结合 Linux 资源观察、服务日志和 MySQL 慢查询去分析瓶颈。

### 6. 性能测试中你通常关注哪些指标

**答案：**

- 响应时间
- 吞吐量
- 错误率
- QPS / TPS

### 7. 你们做性能测试的通过标准是什么

**答案：**

- 性能测试是否通过，一般要看核心接口或核心链路是否满足预设指标。
- 常见会综合看响应时间、吞吐量、错误率、资源占用是否在可接受范围内。
- 同时也要看压测过程中是否出现明显瓶颈，比如慢查询、资源打满、错误率持续升高等。

### 8. 性能测试常见哪些性能问题

**答案：**

- 响应时间过长
- 吞吐量上不去
- 错误率升高
- CPU、内存、磁盘 IO 资源瓶颈
- 数据库慢查询
- 线程、连接、并发处理能力不足

### 9. 你怎么理解性能瓶颈

**答案：**

- 性能瓶颈可能出现在应用层、数据库层、网络层，也可能出现在压测脚本自身。
- 所以定位时不能只看 JMeter 报表，还要结合 Linux 资源、服务日志和慢查询一起看。

## 七、计算机网络

### 1. TCP 和 UDP 的区别

**答案：**

- TCP 协议可靠，保证数据安全到达，也保证有序。
- UDP 协议不可靠，不保证数据安全到达，也不保证顺序。
- TCP 传输速度相对慢，资源开销大。
- UDP 传输速度快，资源开销比 TCP 小。
- TCP 适合对数据安全敏感的场景，例如登录功能。
- UDP 适合对实时性要求较高的场景，例如音视频通话、网络游戏。

### 2. HTTP 和 TCP 之间的关系

**答案：**

- HTTP 是基于 TCP 实现的。
- HTTP 是应用层协议，TCP 是传输层协议。

### 3. 描述一次 HTTP 请求过程

**答案：**

- 在浏览器地址栏输入 URL 后，通过 TCP/IP 协议向服务器发起请求。
- 先经过三次握手建立连接。
- 服务器把资源传输给浏览器。
- 传输结束后再经过四次挥手断开客户端和服务端连接。

### 4. HTTP 和 HTTPS 的区别

**答案：**

- HTTP 明文传输。
- HTTPS 使用 SSL/TLS 加密，安全性更高。
- HTTP 一般是 80 端口，HTTPS 一般是 443 端口。

### 1. Python 中列表和元组的区别

### 2. Python 中怎么传可变参数，`*` 和 `**` 是什么意思

- `*` 一般表示位置可变参数。
- `**` 一般表示关键字可变参数。

### 3. 如何捕捉一个异常

### 4. 浅拷贝和深拷贝的区别

**答案：**

- 浅拷贝复制对象顶层引用。
- 深拷贝会递归复制所有子对象。

### 5. 用 Python 如何实现冒泡排序

**答案：**

```python
def bubble_sort(nums):
    n = len(nums)
    for i in range(n - 1):
        for j in range(n - 1 - i):
            if nums[j] > nums[j + 1]:
                nums[j], nums[j + 1] = nums[j + 1], nums[j]
    return nums
```

### 6. 什么是装饰器

**答案：**

- 装饰器我的理解就是修饰函数的函数。
- 不改变原有函数代码的基础上，动态改变函数功能。
- 在我的框架中，我对装饰器的使用一般是调用已有的，比如 `@pytest.mark.run()`、`@pytest.mark.flaky()`、`@pytest.fixture` 等。

### 7. 什么是可迭代对象、迭代器、生成器

**答案：**

- 可迭代对象一般是能直接用于 `for` 循环遍历的数据对象，比如 `str`、`list`、`tuple`、`set`、`dict` 等。
- 迭代器是可以被 `next()` 调用并不断返回下一个值的对象。
- 生成器是一种特殊的迭代器，通过 `yield` 关键字创建，每次执行到 `yield` 会暂停并保存当前运行状态。
