# 性能测试笔记（整理版）

> 来源：网页内容通过 cpdown 导出后整理。已去除评论区、推荐区、头像链接等噪音内容。

## 概念

- 性能测试：通过工具或代码模拟不同负载，对系统性能指标进行评估。
- 目的：评估系统能力、定位瓶颈、验证是否满足业务高峰和未来增长需求。
- 常见类型：负载测试、压力测试、稳定性测试、容量测试。

## 问题

- 如何明确性能需求并转成可执行方案？
- 关键指标看哪些？如何判断是否达标？
- 出现响应变慢、错误率上升时如何定位瓶颈？

## 原因

- 并发上来后，后端服务可能进入超负荷状态。
- 数据库慢查询、锁竞争、连接池不足导致请求堆积。
- JVM 内存管理不当可能引发内存泄漏。
- 压测机本身能力不足会造成测试数据失真。

## 解决方式

### 1) 需求与范围

- 明确核心业务（优先保证核心链路性能）。
- 需求来源可分客户侧与内部侧，统一量化为可验收指标。

### 2) 关键指标

- 平均响应时间
- 吞吐量
- QPS / TPS
- 错误率
- 服务器资源：CPU、内存、磁盘、网络

### 3) 测试流程

1. 功能测试通过后进入性能测试
2. 需求分析
3. 测试计划
4. 用例设计
5. 环境搭建
6. 执行测试
7. 报告输出
8. 结果分析与调优

### 4) 场景设计

- 单接口性能测试：
  - 同步定时器模拟高并发
  - 常数吞吐量定时器模拟高频率
  - 阶梯式加压观察拐点
- 混合业务性能测试：按真实业务比例混合核心操作。
- 分布式性能测试：单机线程不足时，多机协同压测。

## 示例

### WMS 实战案例 1：并发下唯一校验失效

- 场景：核心业务“创建入库单”。
- 现象：并发新增时，唯一校验失效。
- 定位：怀疑并发数据处理策略，和开发沟通后确认并发竞争问题。
- 解决：对表数据加锁，复测后问题修复。

### WMS 实战案例 2：高峰期内存泄漏

- 业务背景：每周一约 2 小时高峰，约 100 名跟单员高频操作。
- 压测模型：100 用户，平均每秒各请求 1 次，混合“查询入库单/入库/编辑/导出”。
- 现象：发现内存泄漏。
- 定位：导出功能使用文件流式写入但未关闭写入流。
- 解决：补充关闭流代码，复测恢复正常。

## 面试表达

### 常见性能瓶颈

1. 压测机瓶颈：JMeter 单机负载能力有限，超出后会卡顿。
2. 程序机制瓶颈：业务复杂或算法低效导致处理变慢。
3. JVM 瓶颈：内存未及时释放，导致泄漏或频繁 GC。
4. 数据库瓶颈：慢查询、索引不合理、连接池过小。
5. 服务器资源瓶颈：CPU/内存/磁盘/网络达到上限。

### 概念补充

- 集合点：JMeter 中常用同步定时器实现并发聚合触发。
- 检查点：本质是断言，用于校验接口返回是否符合预期。

### 可直接复述的回答框架

> 我做性能测试会先明确核心业务和指标，再按单接口、混合业务、分布式逐层压测。执行过程中同时看响应时间、吞吐量、错误率和资源占用。出现问题后先区分是压测机、应用、JVM 还是数据库瓶颈，再通过日志、监控和 SQL 排查定位，最后调优后复测闭环。
