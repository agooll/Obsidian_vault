# Auto_prd_test_expert：项目运行逻辑与代码跳转阅读法

## 这篇笔记解决什么问题

看这种项目，最难的通常不是 Python 语法，而是：

1. 框架自己的 API 在干什么
2. 一个函数为什么会跳到另一个文件
3. 哪条调用链才是主流程，哪些只是辅助逻辑

所以阅读时不要按文件顺序硬读，而要按“运行路径”读。

---

## 一条核心原则

看项目时，不要问“这个文件里写了什么”，先问：

> 这个文件在程序运行时，是在哪一步被调用的？

也就是从入口开始，顺着调用链往下追：

`ui/main.py -> ui/sidebar.py / core/llm_client.py / core/rag_engine.py / core/evaluator.py`

这样你就不会在 API 和函数跳转里迷路。

---

## 怎么区分三种代码

每一行代码，先判断它属于下面哪一类：

### 1. 页面控制代码
特点：通常是 `st.xxx`

例子：
- `st.set_page_config(...)`
- `st.button(...)`
- `st.file_uploader(...)`
- `st.tabs(...)`

含义：这些是 Streamlit 提供的 API，用来画页面和处理交互。

### 2. 项目自己的业务代码
特点：函数名、类名能在当前项目里搜到定义

例子：
- `render_sidebar()`
- `get_gemini_chat_response()`
- `RAGEngine(...)`
- `Evaluator(...)`

含义：这些是真正的项目逻辑。

### 3. 第三方库能力
特点：一般是某个库对象调用的方法

例子：
- `genai.list_models()`
- `model.generate_content(...)`
- `chromadb.PersistentClient(...)`
- `pd.DataFrame(...)`

含义：这些不是项目作者自己造的，而是外部 SDK 或框架提供的能力。

## 这个项目的主入口在哪里

主入口在：`ui/main.py`

关键点：
- `main()` 是页面逻辑起点
- `if __name__ == "__main__": main()` 表示文件运行时从这里开始

所以阅读整个项目时，应当先看 `ui/main.py`，不要先看 `core/`。

---

## 这个项目的主运行链路

把整个项目先压成一条主流程：

1. `main()` 启动
2. `setup_proxy()` 配置代理
3. `render_sidebar()` 读取 API Key 和模型选择
4. `RAGEngine(api_key)` 初始化知识库引擎
5. `Evaluator(api_key)` 初始化评估器
6. 用户上传 PRD / PDF / 图片 / 文本
7. `rag_engine.search_context(...)` 做第一次 RAG 检索
8. `PromptManager.get_rag_filter_prompt(...)` 构造过滤 Prompt
9. `get_gemini_chat_response(...)` 用 Gemini 对召回内容做二次过滤
10. 用户点击“开始生成”
11. `PromptManager.get_initial_prompt(...)` 组装 PRD + RAG 上下文
12. `get_gemini_chat_response(...)` 正式生成测试用例
13. `extract_json_from_text(...)` 从模型返回中提取 JSON
14. `display_results(...)` 展示表格和导出内容
15. `evaluator.evaluate_cases(...)` 做质量评估
16. `rag_engine.add_history_case(...)` 把最终结果归档进历史案例库

你只要能把这 16 步讲顺，这个项目的运行逻辑你就已经抓住了。

## 按文件看：每个文件在主流程里的位置

### 1. `ui/main.py`
职责：总控编排器

它不负责底层算法，主要负责把页面、配置、RAG、模型、评估、展示串起来。

阅读时重点盯住这几个动作：
- 初始化 session_state
- 调 sidebar 拿配置
- 上传文件
- 检索知识库
- 调模型生成
- 展示结果
- 调评估器
- 归档历史案例

### 2. `ui/sidebar.py`
职责：侧边栏配置入口

它负责：
- API Key 输入
- 读取/保存本地配置
- 动态获取可用模型
- 清空工作台

它本身不生成测试用例，只负责把“配置”交给 `main()`。

### 3. `core/llm_client.py`
职责：模型访问层

它负责：
- 获取当前 API Key 可用的模型列表
- 调用 Gemini 进行聊天或生成
- 从模型回复里提取 JSON
- 生成摘要

为什么要单独抽出来：
- UI 不该直接写 SDK 细节
- 以后换模型时改动更集中
- JSON 提取逻辑可复用

### 4. `core/rag_engine.py`
职责：知识库引擎

它负责：
- 连接 ChromaDB
- 保存上传的原始文件
- 文本切片
- 文档入库
- 历史案例入库
- 检索知识库上下文
- 删除和查看知识文档

它是“RAG 能力”的核心文件。

### 5. `core/evaluator.py`
职责：质量评估器

它负责：
- 根据 PRD、当前生成结果、RAG 规范、标准答案
- 调模型输出质量报告
- 返回结构化评分、漏测点、逻辑问题、重复项、优化建议

它不是生成器，而是“验收器”。

### 6. `config/prompts.py`
职责：统一管理 Prompt

它负责：
- 主生成 Prompt
- 摘要 Prompt
- 多模态解析 Prompt
- 评估 Prompt
- RAG 过滤 Prompt

这说明项目的行为规则并不全写死在业务代码里，很多是通过 Prompt 配置出来的。

### 7. `config/settings.py`
职责：运行配置层

它负责：
- 代理设置
- 路径配置
- 读取 `data/user_config.json`
- 从环境变量读取 `GEMINI_API_KEY`

它不参与测试用例生成本身，但属于运行底座。

## 看“跳来跳去”的代码时，怎么不晕

你以后看任意函数，按这个顺序拆：

### 第一步：先看是谁调用了它
不要一上来就研究函数内部细节。

先问：
- 这个函数在哪个入口里被用到？
- 它在主流程第几步？

### 第二步：再看它返回什么
重点看：
- 输入参数是什么
- 返回值是什么
- 返回值被谁接住了

因为函数之间真正的连接点，不是“文件”，而是“参数和返回值”。

### 第三步：最后才看函数内部细节
如果内部调了第三方 SDK，再单独判断：
- 这是页面 API？
- 这是项目函数？
- 这是外部库 API？

这样阅读会稳定很多。

## 一个具体例子：从 `render_sidebar()` 往下追

在 `ui/main.py` 里：

```python
api_key, selected_model = render_sidebar()
```

你应该这样理解：

1. `render_sidebar()` 在 `ui/sidebar.py` 定义
2. 它的任务是渲染侧边栏
3. 它最后返回两个值：`api_key` 和 `selected_model`
4. `main()` 接住这两个值后，后面初始化 `RAGEngine` 和 `Evaluator` 都要靠它们

所以 `render_sidebar()` 的关键不是“画了什么控件”，而是：

> 它给主流程提供了后续运行所需的配置输入。

这就是“从调用关系看函数”的思路。

---

## 一个具体例子：从 `get_gemini_chat_response()` 往下追

在 `ui/main.py` 里，当用户点击生成时，会调：

```python
resp_text, updated_history = get_gemini_chat_response(...)
```

这里你要抓的重点是：

1. 输入：API Key、模型名、历史消息、Prompt
2. 输出：模型回复文本、更新后的对话历史
3. 后续：
   - `resp_text` 被拿去提取 JSON
   - `updated_history` 被存进 `st.session_state['gemini_history']`

所以这个函数的角色不是“随便调一下模型”，而是：

> 负责承接多轮对话生成，并把新的模型上下文返回给页面层保存。

## 你在 Obsidian 里应该怎么记笔记

每读一个文件，只记 4 件事：

```md
## 文件名
### 1. 这个文件的职责
一句话说明它干什么

### 2. 对外暴露了哪些函数/类
- 函数A：做什么
- 函数B：做什么

### 3. 它依赖谁
- 调了哪个项目内函数
- 调了哪个第三方 API

### 4. 它被谁调用
- 从哪个入口进来
- 在主流程的第几步被用到
```

这比你记一大堆零散代码细节有用得多。
